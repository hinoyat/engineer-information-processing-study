### 001. 소프트웨어 생명 주기 (SDLC)

#### 📘 정의
소프트웨어 생명 주기(SDLC, Software Development Life Cycle)란, 소프트웨어의 개발 시작부터 유지보수 및 폐기까지의 전 과정을 체계적으로 정의한 절차이다.  
각 단계는 프로젝트의 명확한 진행과 품질 관리를 위해 구성된다.

#### 🧩 핵심 개념 요약 및 비교표
- SDLC는 **계획 → 분석 → 설계 → 구현 → 테스트 → 유지보수**의 절차로 구성됨
- 각 단계는 **명확한 산출물과 목표**를 가진다
- 대표적인 SDLC 구조는 **폭포수 모형(Waterfall Model)**

| 단계     | 주요 내용                                      |
|--------|--------------------------------------------|
| 계획     | 요구사항 수집, 전체 일정 수립, 예산·인력 등 자원 계획 수립             |
| 분석     | 요구사항 상세 분석, 논리적 구조 정의, 명세서 작성 등                    |
| 설계     | 시스템 아키텍처 설계, DB 설계, UI 설계 등 기술적 구조 설계 수행         |
| 구현     | 실제 코드 작성 및 모듈별 단위 테스트 수행                         |
| 테스트    | 통합 테스트, 시스템 테스트, 사용자 수용 테스트 등을 통해 품질 검증         |
| 유지보수   | 운영 중인 시스템의 버그 수정, 기능 개선, 환경 변경에 대한 적응 등 수행     |

#### 📝 기출 포맷 예시
- 다음 중 SDLC 단계에 해당하지 않는 것은?
- 소프트웨어 생명 주기의 “설계” 단계에서 수행하는 작업은 무엇인가?
- SDLC에서 “유지보수” 단계의 주요 목적은 무엇인가?

#### 🧠 용어 설명
- **SDLC**: Software Development Life Cycle, 소프트웨어 생명 주기
- **산출물**: 각 단계별 결과물 (예: 요구사항 명세서, 설계서, 테스트 계획서 등)
- **품질 관리**: 소프트웨어의 기능, 성능, 유지보수성을 목표 기준에 맞추는 활동

### 002. 나선형 모형 (Spiral Model)

#### 📘 정의
나선형 모형은 프로토타입 개발을 기반으로 위험 분석을 강화한 반복적 소프트웨어 개발 모델이다.  
**점진적 개발 모델 + 위험 관리 중심 접근법**이 결합된 형태로, 반복(iteration)마다 점차 기능과 범위를 확장한다.

#### 🧩 핵심 개념 요약 및 비교표
- 반복 단위로 계획 → 위험 분석 → 개발 및 검증 → 고객 평가 수행
- 각 주기를 통해 점차 시스템을 완성해가는 방식
- **리스크 분석(Risk Analysis)**을 중심에 둠
- 변경과 피드백에 유연하게 대응 가능

| 반복 단계     | 설명                                          |
|------------|---------------------------------------------|
| 계획 수립     | 요구사항 수립 및 전략 계획                                |
| 위험 분석     | 기술, 예산, 일정 등의 리스크 요인 식별 및 대응 방안 수립               |
| 개발 및 검증   | 기능 개발 및 프로토타입 제작, 검증 테스트 수행                    |
| 고객 평가     | 결과물에 대한 고객 피드백 수렴, 다음 반복 계획 반영                  |

#### 📝 기출 포맷 예시
- 나선형 모형의 핵심 요소로 올바른 것은?
- 나선형 모형에서 "위험 분석" 단계의 역할은?
- 나선형 모델이 적합한 프로젝트 유형은?

#### 🧠 용어 설명
- **Risk Analysis**: 프로젝트 진행 시 발생 가능한 실패 요인을 사전에 식별하고 대비하는 과정
- **프로토타입**: 요구사항을 반영한 시험용 소프트웨어 시제품
- **Iteration**: 반복 주기로 시스템을 점진적으로 확장하는 개발 방식

### 003. 폭포수 모형 (Waterfall Model)

#### 📘 정의
폭포수 모형은 소프트웨어 개발 과정에서 각 단계를 **순차적으로** 진행하는 전통적 개발 모델이다.  
이전 단계로 **되돌아갈 수 없는 구조**이며, 한 단계를 마무리하고 승인한 후 다음 단계로 넘어가는 방식이다.  
문서화 중심 관리로 품질과 책임 소재를 명확히 하며, **요구사항이 명확한 프로젝트**에 적합하다.

#### 🧩 핵심 개념 요약 및 비교표
- 각 단계는 명확히 구분되고, 상위 단계 종료 후 하위 단계 착수
- 변경에 대한 유연성이 낮고, 사용자의 피드백 반영이 어렵다
- 문서화가 잘 되어 있어 품질 보증 및 외주 관리에 유리

| 장점                             | 단점                                 |
|--------------------------------|------------------------------------|
| 단계별 문서화로 품질 관리 용이              | 변경에 유연하지 못함                       |
| 일정 및 비용 예측이 쉬움                 | 요구사항이 명확하지 않으면 실패 가능성 높음         |
| 이해 및 구현이 비교적 쉬움               | 사용자 피드백 반영이 어려움                   |

#### 📝 기출 포맷 예시
- 다음 중 폭포수 모형의 특징으로 올바르지 않은 것은?
- 폭포수 모델이 적합한 상황은 어떤 경우인가?
- 폭포수 모형의 개발 단계 순서로 올바른 것은?

#### 🧠 용어 설명
- **폭포수(Waterfall)**: 위에서 아래로 흐르는 절차적 구조를 비유한 명칭
- **단계별 산출물**: 각 단계에서 문서 등으로 결과를 명확히 남김
- **변경에 비유연**: 초기 계획이 바뀌기 어렵고, 되돌리기 어려운 구조

### 004. 애자일 모형 (Agile Model)

#### 📘 정의
애자일(Agile) 모형은 고객의 요구사항 변화에 **민첩하게 대응**하며,  
**반복적이고 점진적으로** 소프트웨어를 개발하는 방법론이다.  
고객과의 소통, 피드백, 빠른 납품을 중심으로 하는 유연한 개발 방식이다.

#### 🧩 핵심 개념 요약
- 변화 수용, 빠른 피드백, 지속적 개선
- 문서보다는 작동하는 소프트웨어, 계획보다는 변화 대응을 중시
- 대표 애자일 기법:
    - **스크럼(Scrum)**
    - **XP(eXtreme Programming)**
    - **칸반(Kanban)**
    - **Lean**
    - **기능 중심 개발 - FDD (Feature Driven Development)**

#### 📝 기출 포맷 예시
- 다음 중 애자일 개발 방법론의 특징이 아닌 것은?
- 반복적 개발과 빠른 피드백을 강조하는 개발 방식은?
- 애자일 방법론에 속하는 기법으로 적절한 것은?

#### 🧠 용어 설명
- **애자일**: 변화에 유연하게 적응하며 민첩하게 개발하는 방식
- **반복적 개발**: 기능 단위로 나누어 점진적으로 완성하는 개발 방식
- **스크럼**: 역할과 절차가 명확한 대표적인 애자일 프레임워크

### 005. 애자일 개발 4가지 핵심 가치

#### 📘 정의
애자일 선언문(Agile Manifesto)에 명시된 개발 철학으로,  
소프트웨어 개발 시 어디에 더 큰 가치를 둘 것인지에 대한 기준이다.

#### 🧩 핵심 개념 요약
- **개인과 상호작용 > 프로세스와 도구**
- **작동하는 소프트웨어 > 포괄적인 문서**
- **고객과의 협업 > 계약 협상**
- **변화에의 대응 > 계획을 따르기**

#### 📝 기출 포맷 예시
- 다음 중 애자일의 4대 핵심 가치에 해당하지 않는 것은?
- 애자일 선언문의 핵심 가치 중 올바른 조합을 고르시오.

#### 🧠 용어 설명
- **애자일 선언문**: 2001년 개발자들이 발표한 소프트웨어 개발 철학
- **핵심 가치**: 개발 우선순위를 나타내는 네 가지 원칙

### 006. 소프트웨어 공학 (Software Engineering)

#### 📘 정의
소프트웨어 공학은 체계적이고 공학적인 접근을 통해  
소프트웨어를 **효율적으로 개발, 운영, 유지보수**하는 학문이다.  
목표는 소프트웨어의 품질 향상과 생산성 증대이다.

#### 🧩 핵심 개념 요약
- 공학적 원리를 통해 소프트웨어 개발의 복잡성과 실패율을 줄인다
- 소프트웨어 공학의 기본 원칙:
  - 현대적 기술을 **지속적으로 적용**해야 한다
  - 개발된 SW의 품질을 **지속적으로 검증**해야 한다
  - 개발 결과와 활동을 **명확히 기록**해야 한다

#### 📝 기출 포맷 예시
- 소프트웨어 공학의 기본 원칙으로 올바르지 않은 것은?
- 소프트웨어 공학이 필요한 이유는?

#### 🧠 용어 설명
- **공학적 접근**: 체계적, 논리적, 반복적인 개발 절차
- **소프트웨어 공학**: 소프트웨어 개발의 품질과 효율을 높이기 위한 학문

### 007. 스크럼 개발 프로세스

#### 📘 정의
스크럼(Scrum)은 반복적이고 점진적인 개발을 위한 애자일 방법론 중 하나로,  
**고정된 기간 내 목표를 설정하고 자율적으로 팀이 협력**해 기능을 개발하는 방식이다.  
짧은 반복 주기인 **스프린트(Sprint)**를 중심으로 개발이 이루어진다.

#### 🧩 핵심 개념 요약

| 프로세스 단계 | 설명 |
| ------------- | ---- |
| **스프린트 계획 회의** | 개발 목표 설정, 우선순위 선정, 역할 분담 등 스프린트 준비 |
| **스프린트(Sprint)** | 정해진 기간(보통 2~4주) 동안의 실제 개발 작업 수행 |
| **일일 스크럼 회의** | 매일 15분 내외로 진행, 진행 상황과 장애 요인 공유 |
| **스프린트 검토 회의** | 결과물을 시연하며 고객 또는 이해관계자 피드백 수렴 |
| **스프린트 회고 회의** | 프로세스 개선점 도출, 다음 스프린트 개선 방향 논의 |

#### 📝 기출 포맷 예시
- 다음 중 스크럼의 일일 회의에서 다루지 않는 항목은?
- 스크럼 프로세스의 순서를 올바르게 나열한 것은?

#### 🧠 용어 설명
- **스프린트**: 정해진 기간 내에 수행하는 개발 단위
- **일일 스크럼**: 진행 상황, 장애 요인, 다음 작업 공유 회의
- **회고(Retrospective)**: 회차별 문제점 개선을 위한 피드백 시간

### 008. XP (eXtreme Programming)

#### 📘 정의
XP는 고객의 요구사항 변화에 유연하게 대응하기 위해,  
**극단적인 개발 모범 사례**를 실천하는 애자일 방법론이다.  
소규모 팀에서 적용이 용이하고, **빠른 릴리즈와 지속적 통합**을 중시한다.

#### 🧩 핵심 개념 요약
- XP의 5가지 핵심 가치:
  - **의사소통(Communication)**
  - **단순성(Simplicity)**
  - **용기(Courage)**
  - **존중(Respect)**
  - **피드백(Feedback)**

#### 📝 기출 포맷 예시
- 다음 중 XP의 핵심 가치에 포함되지 않는 것은?
- XP 방법론의 특징으로 올바른 것은?

#### 🧠 용어 설명
- **XP**: 극단적인 개발 기법을 반복적, 실천적으로 적용하는 애자일 방법론
- **핵심 가치**: 팀워크와 유연한 개발을 위한 행동 철학

---

### 009. XP의 주요 실천 방법 (Practice)

#### 📘 정의
XP의 실천 방법은 **핵심 가치를 실현하기 위한 구체적인 개발 기법**이다.  
모든 구성원이 참여하고, 지속적인 테스트와 통합을 통해 품질을 확보한다.

#### 🧩 핵심 실천 항목

- **Pair Programming (짝 프로그래밍)**: 두 명이 한 컴퓨터로 함께 코딩
- **Collective Ownership (공동 코드 소유)**: 모든 팀원이 모든 코드에 책임
- **Test-Driven Development (TDD)**: 테스트 먼저 작성, 그다음 실제 코드 작성
- **Whole Team (전체 팀 참여)**: 고객 포함 전체 팀원이 개발에 관여
- **Continuous Integration (지속적 통합)**: 코드 변경 시마다 통합 및 자동 테스트 수행
- **Refactoring (리팩토링)**: 기능 변화 없이 구조를 개선하여 유지보수성 향상
- **Small Releases (소규모 릴리즈)**: 빠르고 자주 릴리즈하여 고객 피드백 반영

#### 📝 기출 포맷 예시
- 다음 중 XP 실천 기법이 아닌 것은?
- 테스트 주도 개발(TDD)의 특징으로 올바른 것은?

#### 🧠 용어 설명
- **TDD**: 테스트 코드를 먼저 작성한 뒤 실제 코드 작성
- **Refactoring**: 내부 구조 개선, 유지보수성 향상
- **CI**: 지속적 통합을 통해 안정된 빌드 유지

### 010. 데이터베이스 관리 시스템 (DBMS)

#### 📘 정의
DBMS(DataBase Management System)는  
사용자와 데이터베이스 사이에서 데이터를 효율적으로 관리하고  
정보를 생성·저장·검색·갱신·삭제할 수 있도록 지원하는 **소프트웨어 시스템**이다.

#### 🧩 핵심 개념 요약
- DBMS는 **데이터의 무결성, 보안, 일관성, 공유성**을 보장한다.
- 사용자 요청을 처리하며, **데이터 저장, 질의 처리, 트랜잭션 관리** 등을 수행한다.
- 요구사항 분석 시 다음 사항들을 고려해야 한다:

| 고려 요소 | 설명 |
| --- | --- |
| **가용성** | 데이터 접근 가능성과 시스템의 가동 시간 |
| **성능** | 응답 시간, 처리량 등 시스템 효율성 |
| **기술 지원** | 벤더 지원, 문서화 수준 등 |
| **상호 호환성** | 다른 시스템과의 연계 가능 여부 |
| **구축 비용** | 도입, 운영, 유지보수에 필요한 비용 |

#### 📝 기출 포맷 예시
- DBMS 선정 시 고려사항으로 옳지 않은 것은?
- DBMS의 주요 기능으로 볼 수 없는 것은?

#### 🧠 용어 설명
- **무결성**: 데이터의 정확성과 일관성 유지
- **트랜잭션**: DB 작업의 최소 단위, 원자성 보장
- **질의(Query)**: 사용자의 데이터 요청

---

### 011. 기능 요구사항

#### 📘 정의
기능 요구사항(Functional Requirements)은  
시스템이 **무엇을 해야 하는지**, 즉 **사용자가 어떤 기능을 수행할 수 있어야 하는지**를 명시한 요구사항이다.

#### 🧩 핵심 개념 요약
- 시스템이 제공해야 할 **기능, 서비스, 동작**을 정의함
- 사용자의 **입력/출력 흐름**, **처리 방식**, **저장 및 검색 또는 연산 기능** 등 포함
- 일반적으로 명확하게 정의 가능하고, **테스트 가능**함
- 사용자가 제공받기 원하는 기능

#### 📝 기출 포맷 예시
- 다음 중 기능 요구사항에 해당하는 것은?
- 기능 요구사항과 비기능 요구사항을 구분하시오.

#### 🧠 용어 설명
- **기능 요구사항**: 시스템의 주요 동작 및 행위 정의
- **출력 조건**: 기능의 결과로 제공해야 하는 정보

---

### 012. 비기능 요구사항

#### 📘 정의
비기능 요구사항(Non-functional Requirements)은  
시스템이 **어떻게 동작해야 하는지**에 대한 요구사항으로,  
**성능, 보안, 가용성, 호환성 등 품질 특성**을 정의한다.

#### 🧩 핵심 개념 요약

| 분류 | 설명 |
| --- | --- |
| **성능** | 응답 시간, 처리량, 자원 효율성 등 |
| **보안** | 접근 제어, 데이터 보호 |
| **가용성** | 시스템 다운타임 최소화 |
| **사용성** | 인터페이스의 직관성, 학습 난이도 |
| **이식성** | 다른 환경에서도 실행 가능 |
| **유지보수성** | 시스템의 변경, 확장 용이성 |
| **제약사항** | 특정 기술, 예산, 일정 제한 등

#### 📝 기출 포맷 예시
- 다음 중 비기능 요구사항에 해당하지 않는 것은?
- 성능, 보안 등의 품질 특성은 어떤 요구사항인가?

#### 🧠 용어 설명
- **품질 요구사항**: 성능, 안정성, 확장성 등 비기능 요소
- **제약사항**: 프로젝트 실행 시 고려해야 할 외부 조건

---

### 013. 요구사항 명세 기법

#### 📘 정의
요구사항 명세 기법은 **사용자의 요구를 문서화하고 개발자가 이해할 수 있도록 표현**하는 방법이다.  
정형/비정형 기법으로 구분되며, 사용 목적과 상황에 따라 선택된다.

#### 🧩 명세 기법 비교표

| 구분 | 정형 기법 | 비정형 기법 |
|------|-----------|-------------|
| 특징 | 수학적 기반, 명확하고 일관성 있음 | 자연어 기반, 직관적이나 모호함 |
| 장점 | 오류 예방, 자동 분석 가능 | 이해 쉬움, 작성 간편 |
| 단점 | 작성 복잡, 학습 필요 | 해석 차이 발생 가능 |
| 예시 | VDM, Z, Petri-net, CSP 등 | FSM, ERD, 상태 다이어그램, SADT 등 |

#### 📝 기출 포맷 예시
- 다음 중 정형 명세 기법에 해당하는 것은?
- 요구사항 명세 기법 중 자연어 기반으로 이해하기 쉬운 것은?

#### 🧠 용어 설명
- **정형 기법(Formal Method)**: 수학적 표현을 기반으로 요구사항을 기술하는 방법
- **비정형 기법(Informal Method)**: 자연어, 도표 등 직관적 수단을 활용한 방법

---

### 014. 요구사항 개발 프로세스

#### 📘 정의
요구사항 개발 프로세스는 **개발 대상에 대한 요구사항을 도출, 분석, 명세화, 확인**하는 활동이다.  
이를 통해 시스템이 충족해야 할 조건을 구체화하고, **요구공학(Requirement Engineering)**이라 부르기도 한다.
타당성 조사(Feasibility Study)가 먼저 선행되어야 한다.

#### 🧩 핵심 단계 요약

| 단계 | 설명 |
|------|------|
| **도출(Elicitation)** | 사용자 인터뷰, 회의 등을 통해 요구 수집 |
| **분석(Analysis)** | 요구사항 간 충돌·중복·누락 검토 및 정제 |
| **명세화(Specification)** | 문서화, 정형화하여 이해 가능하도록 표현 |
| **확인(Validation)** | 요구사항이 실제 의도를 충족하는지 검토 및 승인 |

#### 📝 기출 포맷 예시
- 요구사항 개발 단계 중 순서로 올바른 것은?
- 요구사항 분석에 해당하지 않는 단계는?

#### 🧠 용어 설명
- **요구공학(Requirement Engineering)**: 요구사항을 다루는 전체 활동
- **명세서(Specification Document)**: 요구사항을 형식적으로 기록한 문서

---

### 015. 요구사항 분석

#### 📘 정의
요구사항 분석(Requirement Analysis)은  
**사용자의 요구를 이해하고, 실현 가능성·비용·일정 등을 고려하여 문서화하는 실질적 분석 단계**이다.  
개발 대상 시스템의 범위와 목적을 명확히 하기 위해 수행된다.

#### 🧩 핵심 개념 요약

- 사용자 요구사항을 **구체적이고 명확하게 추출**
- 요구사항 간 **충돌, 중복, 누락 여부 분석**
- 시스템 구축 시 고려해야 할 **비용/일정/기술 가능성 평가 포함**
- 결과적으로 **기능 목록과 우선순위**를 정의

#### 📝 기출 포맷 예시
- 요구사항 분석 단계에서 수행하는 작업은 무엇인가?
- 소프트웨어 개발 과정 중 분석에 해당하는 것은?

#### 🧠 용어 설명
- **타당성 분석**: 요구가 현실적이고 실현 가능한지 검토하는 과정
- **범위 정의(Scope Definition)**: 개발 범위를 명확히 규정하는 작업

---

### 016. 자료 흐름도 (DFD: Data Flow Diagram)

#### 📘 정의
자료 흐름도(DFD)는 시스템의 **처리 과정, 데이터 흐름, 저장소, 외부 개체 간의 관계를 시각적으로 표현한 다이어그램**이다.  
요구사항 분석 단계에서 시스템의 논리적 구조를 도식화하는 데 사용된다.
자료 흐름과 처리를 중심으로 하는 구조적 분석 기법에 이용

#### 🧩 핵심 개념 요약
- 시스템의 **논리적 흐름**을 표현, 구현과 독립적
- **데이터가 어떻게 이동하고 처리되는지**를 시각적으로 보여줌
- 일반적으로 **계층적 구조**로 설계됨 (상위 → 하위 상세화)

| 구분 | 설명 |
|------|------|
| **물리적 DFD** | 실제 시스템 구성요소(장비, 사용자, DB 등) 중심 표현 |
| **논리적 DFD** | 처리 기능과 데이터 흐름 중심, 구현 독립적 표현 |

#### 📝 기출 포맷 예시
- 자료 흐름도의 목적과 설명으로 옳지 않은 것은?
- 논리적 DFD와 물리적 DFD의 차이를 묻는 문제

#### 🧠 용어 설명
- **DFD**: Data Flow Diagram, 자료 흐름도
- **계층적 설계**: 상위 수준에서 시작해 점진적으로 상세화하는 방식

---

### 017. 자료 흐름도의 구성 요소

#### 📘 정의
자료 흐름도(DFD)는 총 **4가지 구성 요소**로 시스템의 흐름을 시각화한다.  
각 구성 요소는 **고유한 기호와 명확한 역할**을 가지며,  
**요구사항 분석 단계에서 시스템 구조를 정의**하는 데 핵심적인 도구로 사용된다.

#### 🧩 구성 요소 요약

| 구성 요소 | 교재 용어 | 국제 용어 | 기호 | 설명 |
|------------|------------|-------------|------|------|
| 프로세스 | 프로세스 | Process | 원(○) 또는 사각형 내부 번호 | 입력 데이터를 처리하여 출력으로 변환하는 기능 단위 |
| 자료 흐름 | 자료 흐름 | Data Flow | 화살표(→) | 데이터가 시스템 내에서 이동하는 방향 및 흐름 |
| 자료 저장소 | 자료 저장소 | Data Store | 이중선(=) 또는 열린 사각형 | 데이터를 임시 또는 영구적으로 저장하는 장소 |
| 단말 | 단말 | 외부 개체 (External Entity, Terminator) | 사각형 | 시스템 외부에서 데이터를 주고받는 사용자, 타 시스템 등 |

#### ✅ 주요 특징
- **각 요소는 반드시 고유한 명칭 부여**
- **자료 흐름은 반드시 방향이 있어야 함 (양방향은 2개의 화살표로 표현)**
- **프로세스는 최소 1개의 입력과 1개의 출력 흐름을 가져야 함**
- **단말은 DFD 내 시스템 경계를 정의하는 기준점 역할**

#### 📝 기출 포맷 예시
- 다음 중 자료 흐름도(DFD)의 구성 요소가 아닌 것은?
- 각 기호에 해당하는 설명으로 옳지 않은 것은?

#### 🧠 용어 설명
- **프로세스(Process)**: 입력을 받아 처리하고 출력하는 기능 단위
- **자료 흐름(Data Flow)**: 데이터 이동 경로 및 방향
- **자료 저장소(Data Store)**: 데이터가 저장되는 논리적 공간
- **단말(Terminator / External Entity)**: 시스템 외부에서 상호작용하는 사용자 또는 타 시스템

---

### 018. 자료 사전 (DD: Data Dictionary)

#### 📘 정의
자료 사전(Data Dictionary)은 자료 흐름도(DFD)에서 사용되는 **데이터에 대한 상세 정보를 구조적으로 정의**한 문서이다.  
자료명, 구성, 형식, 단위, 출처 등을 명시하여 **시스템 내 데이터의 일관성과 명확성**을 유지한다.

#### 🧩 핵심 개념 요약
- DFD에서 사용된 **자료 흐름, 자료 저장소, 프로세스 입력/출력** 등을 설명
- 자료의 구조, 사용 위치, 유형 등을 정리하여 **요구사항 분석의 정확도 향상**
- **데이터 일관성 유지, 중복 방지, 용어 통일** 등에 활용됨

| 항목 | 설명 예시 |
|------|-----------|
| **자료명** | 회원정보 |
| **자료 구성** | 회원번호 + 이름 + 전화번호 |
| **자료형** | 문자형(Char), 숫자형(Int) 등 |
| **단위** | 명, 건, 개 등 |
| **출처/목적지** | 외부 입력, DB 저장, 화면 출력 등 |

#### 🧩 자료 사전 표현 기호 및 의미

| 기호 | 의미 |
|------|------|
| `=` | 자료 정의 (구성 표시) |
| `+` | 순차적 연결 (AND 관계) |
| `[]` | 선택 (0개 또는 1개 존재 가능) |
| `{}` | 반복 (0개 이상 존재 가능) |
| `()` | 생략 가능 항목 또는 그룹핑 |

#### 📝 기출 포맷 예시
- 자료 사전에 사용되는 기호와 의미 연결이 옳지 않은 것은?
- `{}` 기호는 무엇을 의미하는가?
- 자료 사전의 작성 목적은?

#### 🧠 용어 설명
- **자료 사전(Data Dictionary)**: 데이터 구조를 문서화한 표준 정의서
- **자료 구조 정의**: 여러 개의 하위 항목을 가진 자료를 수식 기호로 표현한 것
- **표현 기호**: 자료 구성에서 사용되는 약속된 구조 기호 (시험 단골 출제 포인트)

---

### 019. 구조적 분석 기법 - SADT (Structured Analysis and Design Technique)

#### 📘 정의
SADT는 **구조적 분석 및 설계 기법**으로, 시스템의 **기능과 데이터 흐름을 계층적으로 분해하여 표현**하는 방법이다.  
요구사항 분석 시 시스템의 동작과 구조를 **명확하게 시각화**하고 **계층적으로 정리**하는 데 사용된다.

#### 🧩 핵심 개념 요약
- **DFD 기반의 상위 분석 기법**으로, 기능 중심 분석을 강조함
- 시스템을 **기능 단위로 나누고**, 각 기능을 **입력/출력/제어/기법**으로 정리
- 주로 **IDEF0 다이어그램** 형태로 표현됨

| 구성 요소 | 의미 |
|-----------|------|
| **입력(Input)** | 처리에 필요한 데이터 |
| **출력(Output)** | 처리 결과로 나오는 데이터 |
| **제어(Control)** | 작업 실행을 제어하는 조건, 규칙 등 |
| **기법(Mechanism)** | 작업 수행을 위한 자원 (사람, 도구 등) |

→ 이 4요소를 ICOM(아이콤) 구성요소라 함

#### 📝 기출 포맷 예시
- SADT 기법에서 제어(Control)의 역할로 올바른 것은?
- SADT의 구성 요소에 포함되지 않는 것은?
- SADT는 어떤 분석 도구로 분류되는가?

#### 🧠 용어 설명
- **SADT**: Structured Analysis and Design Technique
- **ICOM**: Input, Control, Output, Mechanism의 약자
- **계층적 분석**: 상위 기능을 하위 기능으로 나누는 방식

---

### 020. 구조적 분석 기법 - HIPO (Hierarchy Input Process Output)

#### 📘 정의
HIPO는 **계층적 다이어그램을 기반으로 시스템의 입력, 처리, 출력 과정을 구조화**하여 표현하는 분석 기법이다.  
시스템의 전체 기능을 **상위 → 하위로 분해**하면서 시각화하고,  
각 기능별로 **입력(Input), 처리(Process), 출력(Output)** 관계를 문서화한다.

#### 🧩 핵심 개념 요약
- 구조적 분석 기법 중 하나로, **계층적 기능 분할(Hierarchy)**이 핵심
- **히포 차트(HIPO Chart)**와 **IPO 차트**를 함께 사용
- **요구사항을 문서화하고 프로그램 모듈 설계**에 활용 가능

| 구성 요소 | 설명 |
|-----------|------|
| **H (Hierarchy)** | 상위 기능을 하위 기능으로 나누는 계층 구조 |
| **I (Input)** | 기능 수행에 필요한 입력 데이터 |
| **P (Process)** | 입력 데이터를 처리하는 작업 |
| **O (Output)** | 처리 결과로 생성되는 출력 데이터 |

#### 📝 기출 포맷 예시
- HIPO 기법의 구성 요소로 옳지 않은 것은?
- HIPO에서 계층도와 IPO 차트의 목적은?

#### 🧠 용어 설명
- **HIPO**: 계층 구조를 중심으로 입력→처리→출력을 문서화하는 분석 기법
- **히포 차트**: 전체 시스템의 기능 분할 구조를 표현한 다이어그램
- **IPO 차트**: 각 기능의 세부 입력/처리/출력을 정리한 표

### 021. UML (Unified Modeling Language)

#### 📘 정의
UML은 시스템 분석, 설계, 구현 등 **소프트웨어 개발 전 과정에서 시각적으로 모델링**하기 위한  
**객체지향 모델링 언어**이다.  
개발자와 사용자 간의 의사소통을 원활하게 하며,  
복잡한 시스템 구조를 **표준화된 기호와 다이어그램**으로 표현할 수 있게 해준다.

#### 🧩 핵심 개념 요약
- UML은 Rumbaugh(OMT), Booch, Jacobson 등의 방법론을 통합해 등장
- 소프트웨어 개발자 간 **표준화된 설계 언어로 사용됨**
- UML의 3대 구성 요소:

| 요소 | 설명 |
|------|------|
| **사물(Things)** | 모델을 구성하는 실제 객체 또는 개념 (예: 클래스, 인터페이스, 노트 등) |
| **관계(Relationships)** | 사물 간의 연결이나 의존 구조 (예: 연관, 집합, 일반화 등) |
| **다이어그램(Diagram)** | 사물과 관계를 시각적으로 표현한 그림 (정적/동적 모델로 구분됨) |

#### 📝 기출 포맷 예시
- UML의 3대 구성 요소로 옳지 않은 것은?
- UML에서 관계(Relationships)의 종류는?
- 정적 모델과 동적 모델에 해당하는 다이어그램을 구분하시오.

#### 🧠 용어 설명
- **UML**: 객체지향 시스템을 시각화하기 위한 통합 모델링 언어
- **정적 모델링**: 클래스, 객체 등 구조 중심 다이어그램
- **동적 모델링**: 상호작용, 동작 흐름 중심 다이어그램
- **사물(Things)**: 모델 내 실체 (클래스, 객체 등)
- **관계(Relationships)**: 연결, 포함, 상속 등 구조 표현 방식

### 022. 연관 관계 (Association)

#### 📘 정의
연관 관계는 UML에서 **두 개 이상의 클래스(사물)가 서로 관련되어 있는 관계**를 의미한다.  
가장 기본적인 관계이며, 클래스 간의 **실제 연결과 메시지 교환**을 나타낸다.

#### 🧩 핵심 개념 요약
- **실선(—)** 으로 표현하고, **방향성을 갖거나 갖지 않을 수 있음**
- 양방향/단방향 모두 가능
- **다중도(Multiplicity)** 표기로 **각 객체가 몇 개의 인스턴스를 가지는지** 표현

| 항목 | 설명 |
|------|------|
| **표현 방식** | 실선으로 연결 (← 또는 → 방향 표시 가능) |
| **다중도** | `1`, `0..1`, `*`, `1..*` 등 |
| **예시 관계** | "사람 — 집" 관계 (사람은 집을 소유함) |

- 방향을 표시할 경우 **화살표로 표현**, 단방향 관계를 명확히 함
- **다중도는 선 위 또는 양 끝에 표기**

#### 📝 기출 포맷 예시
- UML에서 연관 관계를 표현하는 방법으로 옳은 것은?
- 연관 관계의 다중도 표현으로 올바르지 않은 것은?
- 연관 관계와 관련된 표현 요소가 아닌 것은?

#### 🧠 용어 설명
- **Association (연관 관계)**: 두 클래스가 메시지를 주고받거나 관련되어 있다는 의미
- **다중도(Multiplicity)**: 한 클래스 인스턴스가 상대 클래스와 몇 개나 관계를 가질 수 있는지 표시
- **단방향 연관**: 한쪽 클래스만 상대를 알고 있음
