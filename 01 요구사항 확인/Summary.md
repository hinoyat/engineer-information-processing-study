### 001. 소프트웨어 생명 주기 (SDLC)

#### 📘 정의
소프트웨어 생명 주기(SDLC, Software Development Life Cycle)란, 소프트웨어의 개발 시작부터 유지보수 및 폐기까지의 전 과정을 체계적으로 정의한 절차이다.  
각 단계는 프로젝트의 명확한 진행과 품질 관리를 위해 구성된다.

#### 🧩 핵심 개념 요약 및 비교표
- SDLC는 **계획 → 분석 → 설계 → 구현 → 테스트 → 유지보수**의 절차로 구성됨
- 각 단계는 **명확한 산출물과 목표**를 가진다
- 대표적인 SDLC 구조는 **폭포수 모형(Waterfall Model)**

| 단계     | 주요 내용                                      |
|--------|--------------------------------------------|
| 계획     | 요구사항 수집, 전체 일정 수립, 예산·인력 등 자원 계획 수립             |
| 분석     | 요구사항 상세 분석, 논리적 구조 정의, 명세서 작성 등                    |
| 설계     | 시스템 아키텍처 설계, DB 설계, UI 설계 등 기술적 구조 설계 수행         |
| 구현     | 실제 코드 작성 및 모듈별 단위 테스트 수행                         |
| 테스트    | 통합 테스트, 시스템 테스트, 사용자 수용 테스트 등을 통해 품질 검증         |
| 유지보수   | 운영 중인 시스템의 버그 수정, 기능 개선, 환경 변경에 대한 적응 등 수행     |

#### 📝 기출 포맷 예시
- 다음 중 SDLC 단계에 해당하지 않는 것은?
- 소프트웨어 생명 주기의 “설계” 단계에서 수행하는 작업은 무엇인가?
- SDLC에서 “유지보수” 단계의 주요 목적은 무엇인가?

#### 🧠 용어 설명
- **SDLC**: Software Development Life Cycle, 소프트웨어 생명 주기
- **산출물**: 각 단계별 결과물 (예: 요구사항 명세서, 설계서, 테스트 계획서 등)
- **품질 관리**: 소프트웨어의 기능, 성능, 유지보수성을 목표 기준에 맞추는 활동

### 002. 나선형 모형 (Spiral Model)

#### 📘 정의
나선형 모형은 프로토타입 개발을 기반으로 위험 분석을 강화한 반복적 소프트웨어 개발 모델이다.  
**점진적 개발 모델 + 위험 관리 중심 접근법**이 결합된 형태로, 반복(iteration)마다 점차 기능과 범위를 확장한다.

#### 🧩 핵심 개념 요약 및 비교표
- 반복 단위로 계획 → 위험 분석 → 개발 및 검증 → 고객 평가 수행
- 각 주기를 통해 점차 시스템을 완성해가는 방식
- **리스크 분석(Risk Analysis)**을 중심에 둠
- 변경과 피드백에 유연하게 대응 가능

| 반복 단계     | 설명                                          |
|------------|---------------------------------------------|
| 계획 수립     | 요구사항 수립 및 전략 계획                                |
| 위험 분석     | 기술, 예산, 일정 등의 리스크 요인 식별 및 대응 방안 수립               |
| 개발 및 검증   | 기능 개발 및 프로토타입 제작, 검증 테스트 수행                    |
| 고객 평가     | 결과물에 대한 고객 피드백 수렴, 다음 반복 계획 반영                  |

#### 📝 기출 포맷 예시
- 나선형 모형의 핵심 요소로 올바른 것은?
- 나선형 모형에서 "위험 분석" 단계의 역할은?
- 나선형 모델이 적합한 프로젝트 유형은?

#### 🧠 용어 설명
- **Risk Analysis**: 프로젝트 진행 시 발생 가능한 실패 요인을 사전에 식별하고 대비하는 과정
- **프로토타입**: 요구사항을 반영한 시험용 소프트웨어 시제품
- **Iteration**: 반복 주기로 시스템을 점진적으로 확장하는 개발 방식

### 003. 폭포수 모형 (Waterfall Model)

#### 📘 정의
폭포수 모형은 소프트웨어 개발 과정에서 각 단계를 **순차적으로** 진행하는 전통적 개발 모델이다.  
이전 단계로 **되돌아갈 수 없는 구조**이며, 한 단계를 마무리하고 승인한 후 다음 단계로 넘어가는 방식이다.  
문서화 중심 관리로 품질과 책임 소재를 명확히 하며, **요구사항이 명확한 프로젝트**에 적합하다.

#### 🧩 핵심 개념 요약 및 비교표
- 각 단계는 명확히 구분되고, 상위 단계 종료 후 하위 단계 착수
- 변경에 대한 유연성이 낮고, 사용자의 피드백 반영이 어렵다
- 문서화가 잘 되어 있어 품질 보증 및 외주 관리에 유리

| 장점                             | 단점                                 |
|--------------------------------|------------------------------------|
| 단계별 문서화로 품질 관리 용이              | 변경에 유연하지 못함                       |
| 일정 및 비용 예측이 쉬움                 | 요구사항이 명확하지 않으면 실패 가능성 높음         |
| 이해 및 구현이 비교적 쉬움               | 사용자 피드백 반영이 어려움                   |

#### 📝 기출 포맷 예시
- 다음 중 폭포수 모형의 특징으로 올바르지 않은 것은?
- 폭포수 모델이 적합한 상황은 어떤 경우인가?
- 폭포수 모형의 개발 단계 순서로 올바른 것은?

#### 🧠 용어 설명
- **폭포수(Waterfall)**: 위에서 아래로 흐르는 절차적 구조를 비유한 명칭
- **단계별 산출물**: 각 단계에서 문서 등으로 결과를 명확히 남김
- **변경에 비유연**: 초기 계획이 바뀌기 어렵고, 되돌리기 어려운 구조

### 004. 애자일 모형 (Agile Model)

#### 📘 정의
애자일(Agile) 모형은 고객의 요구사항 변화에 **민첩하게 대응**하며,  
**반복적이고 점진적으로** 소프트웨어를 개발하는 방법론이다.  
고객과의 소통, 피드백, 빠른 납품을 중심으로 하는 유연한 개발 방식이다.

#### 🧩 핵심 개념 요약
- 변화 수용, 빠른 피드백, 지속적 개선
- 문서보다는 작동하는 소프트웨어, 계획보다는 변화 대응을 중시
- 대표 애자일 기법:
    - **스크럼(Scrum)**
    - **XP(eXtreme Programming)**
    - **칸반(Kanban)**
    - **Lean**
    - **기능 중심 개발 - FDD (Feature Driven Development)**

#### 📝 기출 포맷 예시
- 다음 중 애자일 개발 방법론의 특징이 아닌 것은?
- 반복적 개발과 빠른 피드백을 강조하는 개발 방식은?
- 애자일 방법론에 속하는 기법으로 적절한 것은?

#### 🧠 용어 설명
- **애자일**: 변화에 유연하게 적응하며 민첩하게 개발하는 방식
- **반복적 개발**: 기능 단위로 나누어 점진적으로 완성하는 개발 방식
- **스크럼**: 역할과 절차가 명확한 대표적인 애자일 프레임워크

### 005. 애자일 개발 4가지 핵심 가치

#### 📘 정의
애자일 선언문(Agile Manifesto)에 명시된 개발 철학으로,  
소프트웨어 개발 시 어디에 더 큰 가치를 둘 것인지에 대한 기준이다.

#### 🧩 핵심 개념 요약
- **개인과 상호작용 > 프로세스와 도구**
- **작동하는 소프트웨어 > 포괄적인 문서**
- **고객과의 협업 > 계약 협상**
- **변화에의 대응 > 계획을 따르기**

#### 📝 기출 포맷 예시
- 다음 중 애자일의 4대 핵심 가치에 해당하지 않는 것은?
- 애자일 선언문의 핵심 가치 중 올바른 조합을 고르시오.

#### 🧠 용어 설명
- **애자일 선언문**: 2001년 개발자들이 발표한 소프트웨어 개발 철학
- **핵심 가치**: 개발 우선순위를 나타내는 네 가지 원칙

### 006. 소프트웨어 공학 (Software Engineering)

#### 📘 정의
소프트웨어 공학은 체계적이고 공학적인 접근을 통해  
소프트웨어를 **효율적으로 개발, 운영, 유지보수**하는 학문이다.  
목표는 소프트웨어의 품질 향상과 생산성 증대이다.

#### 🧩 핵심 개념 요약
- 공학적 원리를 통해 소프트웨어 개발의 복잡성과 실패율을 줄인다
- 소프트웨어 공학의 기본 원칙:
  - 현대적 기술을 **지속적으로 적용**해야 한다
  - 개발된 SW의 품질을 **지속적으로 검증**해야 한다
  - 개발 결과와 활동을 **명확히 기록**해야 한다

#### 📝 기출 포맷 예시
- 소프트웨어 공학의 기본 원칙으로 올바르지 않은 것은?
- 소프트웨어 공학이 필요한 이유는?

#### 🧠 용어 설명
- **공학적 접근**: 체계적, 논리적, 반복적인 개발 절차
- **소프트웨어 공학**: 소프트웨어 개발의 품질과 효율을 높이기 위한 학문

### 007. 스크럼 개발 프로세스

#### 📘 정의
스크럼(Scrum)은 반복적이고 점진적인 개발을 위한 애자일 방법론 중 하나로,  
**고정된 기간 내 목표를 설정하고 자율적으로 팀이 협력**해 기능을 개발하는 방식이다.  
짧은 반복 주기인 **스프린트(Sprint)**를 중심으로 개발이 이루어진다.

#### 🧩 핵심 개념 요약

| 프로세스 단계 | 설명 |
| ------------- | ---- |
| **스프린트 계획 회의** | 개발 목표 설정, 우선순위 선정, 역할 분담 등 스프린트 준비 |
| **스프린트(Sprint)** | 정해진 기간(보통 2~4주) 동안의 실제 개발 작업 수행 |
| **일일 스크럼 회의** | 매일 15분 내외로 진행, 진행 상황과 장애 요인 공유 |
| **스프린트 검토 회의** | 결과물을 시연하며 고객 또는 이해관계자 피드백 수렴 |
| **스프린트 회고 회의** | 프로세스 개선점 도출, 다음 스프린트 개선 방향 논의 |

#### 📝 기출 포맷 예시
- 다음 중 스크럼의 일일 회의에서 다루지 않는 항목은?
- 스크럼 프로세스의 순서를 올바르게 나열한 것은?

#### 🧠 용어 설명
- **스프린트**: 정해진 기간 내에 수행하는 개발 단위
- **일일 스크럼**: 진행 상황, 장애 요인, 다음 작업 공유 회의
- **회고(Retrospective)**: 회차별 문제점 개선을 위한 피드백 시간

### 008. XP (eXtreme Programming)

#### 📘 정의
XP는 고객의 요구사항 변화에 유연하게 대응하기 위해,  
**극단적인 개발 모범 사례**를 실천하는 애자일 방법론이다.  
소규모 팀에서 적용이 용이하고, **빠른 릴리즈와 지속적 통합**을 중시한다.

#### 🧩 핵심 개념 요약
- XP의 5가지 핵심 가치:
  - **의사소통(Communication)**
  - **단순성(Simplicity)**
  - **용기(Courage)**
  - **존중(Respect)**
  - **피드백(Feedback)**

#### 📝 기출 포맷 예시
- 다음 중 XP의 핵심 가치에 포함되지 않는 것은?
- XP 방법론의 특징으로 올바른 것은?

#### 🧠 용어 설명
- **XP**: 극단적인 개발 기법을 반복적, 실천적으로 적용하는 애자일 방법론
- **핵심 가치**: 팀워크와 유연한 개발을 위한 행동 철학

---

### 009. XP의 주요 실천 방법 (Practice)

#### 📘 정의
XP의 실천 방법은 **핵심 가치를 실현하기 위한 구체적인 개발 기법**이다.  
모든 구성원이 참여하고, 지속적인 테스트와 통합을 통해 품질을 확보한다.

#### 🧩 핵심 실천 항목

- **Pair Programming (짝 프로그래밍)**: 두 명이 한 컴퓨터로 함께 코딩
- **Collective Ownership (공동 코드 소유)**: 모든 팀원이 모든 코드에 책임
- **Test-Driven Development (TDD)**: 테스트 먼저 작성, 그다음 실제 코드 작성
- **Whole Team (전체 팀 참여)**: 고객 포함 전체 팀원이 개발에 관여
- **Continuous Integration (지속적 통합)**: 코드 변경 시마다 통합 및 자동 테스트 수행
- **Refactoring (리팩토링)**: 기능 변화 없이 구조를 개선하여 유지보수성 향상
- **Small Releases (소규모 릴리즈)**: 빠르고 자주 릴리즈하여 고객 피드백 반영

#### 📝 기출 포맷 예시
- 다음 중 XP 실천 기법이 아닌 것은?
- 테스트 주도 개발(TDD)의 특징으로 올바른 것은?

#### 🧠 용어 설명
- **TDD**: 테스트 코드를 먼저 작성한 뒤 실제 코드 작성
- **Refactoring**: 내부 구조 개선, 유지보수성 향상
- **CI**: 지속적 통합을 통해 안정된 빌드 유지

### 010. 데이터베이스 관리 시스템 (DBMS)

#### 📘 정의
DBMS(DataBase Management System)는  
사용자와 데이터베이스 사이에서 데이터를 효율적으로 관리하고  
정보를 생성·저장·검색·갱신·삭제할 수 있도록 지원하는 **소프트웨어 시스템**이다.

#### 🧩 핵심 개념 요약
- DBMS는 **데이터의 무결성, 보안, 일관성, 공유성**을 보장한다.
- 사용자 요청을 처리하며, **데이터 저장, 질의 처리, 트랜잭션 관리** 등을 수행한다.
- 요구사항 분석 시 다음 사항들을 고려해야 한다:

| 고려 요소 | 설명 |
| --- | --- |
| **가용성** | 데이터 접근 가능성과 시스템의 가동 시간 |
| **성능** | 응답 시간, 처리량 등 시스템 효율성 |
| **기술 지원** | 벤더 지원, 문서화 수준 등 |
| **상호 호환성** | 다른 시스템과의 연계 가능 여부 |
| **구축 비용** | 도입, 운영, 유지보수에 필요한 비용 |

#### 📝 기출 포맷 예시
- DBMS 선정 시 고려사항으로 옳지 않은 것은?
- DBMS의 주요 기능으로 볼 수 없는 것은?

#### 🧠 용어 설명
- **무결성**: 데이터의 정확성과 일관성 유지
- **트랜잭션**: DB 작업의 최소 단위, 원자성 보장
- **질의(Query)**: 사용자의 데이터 요청

---

### 011. 기능 요구사항

#### 📘 정의
기능 요구사항(Functional Requirements)은  
시스템이 **무엇을 해야 하는지**, 즉 **사용자가 어떤 기능을 수행할 수 있어야 하는지**를 명시한 요구사항이다.

#### 🧩 핵심 개념 요약
- 시스템이 제공해야 할 **기능, 서비스, 동작**을 정의함
- 사용자의 **입력/출력 흐름**, **처리 방식**, **저장 및 검색 또는 연산 기능** 등 포함
- 일반적으로 명확하게 정의 가능하고, **테스트 가능**함
- 사용자가 제공받기 원하는 기능

#### 📝 기출 포맷 예시
- 다음 중 기능 요구사항에 해당하는 것은?
- 기능 요구사항과 비기능 요구사항을 구분하시오.

#### 🧠 용어 설명
- **기능 요구사항**: 시스템의 주요 동작 및 행위 정의
- **출력 조건**: 기능의 결과로 제공해야 하는 정보

---

### 012. 비기능 요구사항

#### 📘 정의
비기능 요구사항(Non-functional Requirements)은  
시스템이 **어떻게 동작해야 하는지**에 대한 요구사항으로,  
**성능, 보안, 가용성, 호환성 등 품질 특성**을 정의한다.

#### 🧩 핵심 개념 요약

| 분류 | 설명 |
| --- | --- |
| **성능** | 응답 시간, 처리량, 자원 효율성 등 |
| **보안** | 접근 제어, 데이터 보호 |
| **가용성** | 시스템 다운타임 최소화 |
| **사용성** | 인터페이스의 직관성, 학습 난이도 |
| **이식성** | 다른 환경에서도 실행 가능 |
| **유지보수성** | 시스템의 변경, 확장 용이성 |
| **제약사항** | 특정 기술, 예산, 일정 제한 등

#### 📝 기출 포맷 예시
- 다음 중 비기능 요구사항에 해당하지 않는 것은?
- 성능, 보안 등의 품질 특성은 어떤 요구사항인가?

#### 🧠 용어 설명
- **품질 요구사항**: 성능, 안정성, 확장성 등 비기능 요소
- **제약사항**: 프로젝트 실행 시 고려해야 할 외부 조건

---

### 013. 요구사항 명세 기법

#### 📘 정의
요구사항 명세 기법은 **사용자의 요구를 문서화하고 개발자가 이해할 수 있도록 표현**하는 방법이다.  
정형/비정형 기법으로 구분되며, 사용 목적과 상황에 따라 선택된다.

#### 🧩 명세 기법 비교표

| 구분 | 정형 기법 | 비정형 기법 |
|------|-----------|-------------|
| 특징 | 수학적 기반, 명확하고 일관성 있음 | 자연어 기반, 직관적이나 모호함 |
| 장점 | 오류 예방, 자동 분석 가능 | 이해 쉬움, 작성 간편 |
| 단점 | 작성 복잡, 학습 필요 | 해석 차이 발생 가능 |
| 예시 | VDM, Z, Petri-net, CSP 등 | FSM, ERD, 상태 다이어그램, SADT 등 |

#### 📝 기출 포맷 예시
- 다음 중 정형 명세 기법에 해당하는 것은?
- 요구사항 명세 기법 중 자연어 기반으로 이해하기 쉬운 것은?

#### 🧠 용어 설명
- **정형 기법(Formal Method)**: 수학적 표현을 기반으로 요구사항을 기술하는 방법
- **비정형 기법(Informal Method)**: 자연어, 도표 등 직관적 수단을 활용한 방법
