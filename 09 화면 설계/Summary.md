## 🔹 189. UI / UX

#### 📘 정의
- **UI (User Interface)**: 사용자가 컴퓨터 시스템이나 소프트웨어와 **상호작용할 수 있도록 설계된 물리적/논리적 매개체**를 말한다. 입력 장치, 출력 화면, 버튼, 메뉴, 터치 인터페이스 등 사용자가 직접 조작하거나 인식하는 모든 요소가 포함된다.
- **UX (User Experience)**: 사용자가 시스템을 사용하면서 느끼는 **총체적인 경험**을 의미하며, **만족도, 편리성, 감성적 반응**까지 포괄한다.

#### 🧩 UI vs UX 비교

| 구분 | UI (사용자 인터페이스) | UX (사용자 경험) |
|------|--------------------------|--------------------|
| 목적 | 시스템과 상호작용하는 **수단 제공** | 상호작용을 통한 **총체적 만족감 제공** |
| 범위 | 화면 디자인, 버튼, 메뉴 등 **시각적 요소** | 감정, 반응, 사용 흐름 등 **정성적 요소** |
| 구성 | 입력 도구, 출력 장치, 제어 요소 등 | 유용성, 사용성, 효율성, 감성 등 |
| 중심 | **기술적**이고 **디자인 중심** | **경험 중심**의 심리·행동적 접근 |
| 평가 | 사용 편의성, 시각적 완성도 | 만족도, 기대치 충족 여부 등 |

#### 🧩 UI 구성 요소

- **입력 장치**: 키보드, 마우스, 터치스크린 등
- **출력 장치**: 모니터, 프린터, 스피커 등
- **인터페이스 구성요소**: 버튼, 메뉴, 체크박스 등 GUI 요소
- **정보 구조**: 사용자 흐름, 콘텐츠 배치 등

#### 🧩 UX 설계 주요 요소

- **유용성 (Usefulness)**: 사용 목적을 충족시키는 기능 제공 여부
- **사용성 (Usability)**: 쉽게 배우고, 효율적으로 사용할 수 있는지
- **감성 (Desirability)**: 시각적 만족, 브랜드 이미지 등 감정적 반응
- **접근성 (Accessibility)**: 장애 여부에 관계없이 이용 가능한지
- **신뢰성 (Credibility)**: 정보 및 시스템에 대한 신뢰 가능성

#### 📝 기출 포맷 예시

- UI와 UX의 차이를 설명하시오.
- UX의 주요 설계 요소로 옳은 것은?
- 다음 중 UI에 해당하지 않는 것은?

#### 🧠 용어 설명

- **UI**: 사용자와 시스템 간 상호작용을 위한 물리적·논리적 수단
- **UX**: 사용자가 시스템을 이용하면서 느끼는 총체적 경험
- **Usability**: 사용자가 기능을 쉽게 이해하고 효율적으로 사용할 수 있는 정도
- **Desirability**: 제품이 감성적으로 매력적인지를 나타내는 요소

## 🔹 190. UI의 구분

#### 📘 정의
UI의 구분이란, **사용자와 시스템 간의 상호작용 방식에 따라 UI를 유형별로 분류**하는 것으로,  
정보처리기사에서는 **CLI, GUI, NUI** 등의 차이점 및 특징이 자주 출제된다.

#### 🧩 UI 유형별 비교

| 유형 | 명칭 | 특징 | 예시 |
|------|------|------|------|
| **CLI** | Command Line Interface | - 텍스트 기반<br>- 명령어 입력 필요<br>- 학습 곡선 높음 | DOS, Linux Shell |
| **GUI** | Graphical User Interface | - 아이콘, 버튼 등 그래픽 요소<br>- 마우스, 키보드로 조작<br>- 직관적 사용 가능 | Windows, macOS |
| **NUI** | Natural User Interface | - 터치, 음성, 제스처 등 자연스러운 입력 방식<br>- 별도 학습 없이 조작 가능 | Siri, Kinect, 스마트폰 터치 UI |

> 💡 GUI는 시각적 친화성이 높고 실사용 비중도 높아 자주 출제됨  
> 💡 NUI는 **직관성과 감성적 경험 강조**, 최신 인터페이스에서 활용도 증가

#### 🧩 기타 UI 용어

- **VUI (Voice UI)**: 음성으로 조작 (예: AI 스피커)
- **TUI (Touch UI)**: 터치스크린 기반 입력 UI
- **OUI (Organic UI)**: 상황에 따라 형태가 변형되는 인터페이스 (예: 플렉서블 디스플레이)

#### 📝 기출 포맷 예시

- CLI, GUI, NUI의 차이를 설명하시오.
- 다음 중 GUI의 특징으로 옳지 않은 것은?
- 자연스러운 상호작용을 지원하는 UI는?

#### 🧠 용어 설명

- **CLI**: 명령어 기반 텍스트 인터페이스
- **GUI**: 그래픽 기반 시각적 사용자 인터페이스
- **NUI**: 사용자의 제스처, 음성 등 자연스러운 입력 방식
- **VUI**: 음성 인식 기반 UI (예: Siri, Alexa)
- **TUI**: 손가락 또는 스타일러스로 직접 조작하는 UI

## 🔹 191. UI의 기본 원칙

#### 📘 정의
UI의 기본 원칙은 **사용자 중심의 인터페이스 설계를 위해 반드시 고려해야 할 핵심 지침**으로,  
정보처리기사 실기에서는 **직관성, 유효성, 학습성, 유연성 등**의 정의와 구분 문제가 자주 출제된다.

#### 🧩 UI 설계 4대 기본 원칙

| 원칙 | 설명 |
|------|------|
| **직관성** | 사용자가 UI를 보자마자 기능과 동작을 쉽게 이해할 수 있도록 설계 |
| **유효성** | 사용자의 목표 달성에 필요한 기능만을 적절히 제공 |
| **학습성** | UI를 처음 접하는 사용자도 쉽게 배우고 익힐 수 있는 정도 |
| **유연성** | 사용자의 다양한 요구나 상황에 맞게 **입력 방식, 사용 흐름**을 유연하게 수용 |

> 💡 “직유학유”는 암기용 키워드  
> 자주 출제되는 포인트는 “**직관성과 유효성의 차이**” 및 “**유연성 적용 예시**”

#### 🧩 기타 UI 설계 원칙 (출제 빈도 낮음)

| 원칙 | 설명 |
|------|------|
| **가시성(Visibility)** | 현재 상태나 가능한 동작을 명확히 표현 |
| **피드백(Feedback)** | 사용자 조작에 대한 반응을 즉시 제공 |
| **일관성(Consistency)** | 유사한 기능은 동일한 방식으로 제공

#### 📝 기출 포맷 예시

- 다음 중 UI 설계 원칙 중 유효성에 해당하는 설명은?
- 학습성이 높은 UI란 무엇을 의미하는가?
- 직관성과 유연성의 차이를 설명하시오.

#### 🧠 용어 설명

- **직관성**: 별도 설명 없이 누구나 쉽게 이해할 수 있음
- **유효성**: 꼭 필요한 기능만 제공하여 목적 달성 지원
- **학습성**: 사용법을 빠르게 익힐 수 있음
- **유연성**: 다양한 사용자 상황을 수용 가능
- **가시성**: 화면에서 정보와 기능이 잘 보이는 정도
- **피드백**: 사용자 입력에 대한 시스템 반응
- **일관성**: 비슷한 기능은 동일한 UI 방식 적용

## 🔹 192. UI 설계 도구 – 와이어프레임

#### 📘 정의
**와이어프레임(Wireframe)**은 **화면 설계의 뼈대 구조를 나타낸 스케치 형태의 도구**로,  
기능 배치, 콘텐츠 구조, 사용자 흐름 등을 **단순 도형으로 표현**하여 기획자, 디자이너, 개발자 간의 **초기 커뮤니케이션 수단**으로 사용된다.

#### 🧩 주요 특징

- **기능 중심**의 화면 구성 표현 (디자인 요소는 최소화)
- 사용자 흐름 및 콘텐츠 배치 계획 가능
- 이해관계자 간 **의사소통 도구**로 활용
- 화면 전환/링크 흐름도 함께 표현 가능

#### 🧩 구성 요소 예시

- 메뉴 구조
- 버튼 위치
- 입력창/출력 영역 배치
- 화면 흐름도

> 💡 정처기에서는 “와이어프레임은 디자인 요소 없이 기능만 표현”이라는 포인트 자주 출제

#### 📝 기출 포맷 예시

- 와이어프레임의 목적은 무엇인가?
- 다음 중 와이어프레임의 특징으로 옳지 않은 것은?
- 와이어프레임에서 표현하는 정보로 적절한 것은?

#### 🧠 용어 설명

- **와이어프레임**: 콘텐츠 및 기능 배치 중심의 화면 스케치
- **UI 스케치**: 실제 인터페이스 구성을 선 도형으로 표현
- **사용자 흐름도**: 화면 간 이동과 기능 순서 표현 도식

## 🔹 193. 목업 (Mockup)

#### 📘 정의
**목업(Mockup)**은 실제 화면과 유사한 형태로 제작된 **정적인 시안(mock design)**으로,  
와이어프레임보다 구체적이며, **색상, 폰트, UI 디자인 요소** 등을 포함한 **시각적 완성도 높은 화면 설계 도구**이다.

#### 🧩 주요 특징

- **정적인 프로토타입**으로 기능 동작 없이 화면만 구현
- 색상, 그래픽 요소 등 실제 디자인에 가까움
- 사용자 또는 이해관계자에게 **UI 시각화 자료** 제공
- 개발자 및 디자이너 간 **디자인 기준 공유**에 활용

#### 🧩 와이어프레임과 차이

| 구분 | 와이어프레임 | 목업(Mockup) |
|------|---------------|----------------|
| 목적 | 기능 구조 설계 | 시각적 설계 확인 |
| 표현 수준 | 단순 선/구조 중심 | 실제 디자인 반영 |
| 동작 여부 | 없음 | 없음 (정적) |
| 사용 시점 | 기획 초안 단계 | 디자인 확정 전 검토 단계 |

> 💡 목업은 “디자인 요소를 포함하되 동작은 하지 않음”이 핵심 포인트  
> 💡 시각적 품질 확인을 위한 도구라는 점이 자주 출제됨

#### 📝 기출 포맷 예시

- 목업(Mockup)의 특징으로 옳은 것은?
- 와이어프레임과 목업의 차이를 설명하시오.
- 다음 중 목업의 주요 활용 목적이 아닌 것은?

#### 🧠 용어 설명

- **목업(Mockup)**: 실제 화면처럼 구성된 정적인 시안
- **정적 프로토타입**: 동작 없이 디자인만 구현된 모형
- **시각적 요소**: 색상, 레이아웃, 아이콘 등 디자인 구성요소

## 🔹 194. 스토리보드

#### 📘 정의
**스토리보드(Storyboard)**는 **화면 설계의 흐름을 순서대로 나열한 문서**로,  
각 화면에 포함될 **기능, UI 요소, 전환 조건** 등을 설명하며, **기획자·디자이너·개발자 간 의사소통 수단**으로 활용된다.

#### 🧩 주요 특징

- **기획 흐름 기반**으로 각 화면의 **기능, 인터페이스, 전환 방식**을 문서화
- 전체 서비스의 **사용자 시나리오 흐름**을 시각적으로 표현
- 와이어프레임/목업과 함께 사용되어 **개발 명세서 역할** 수행
- 단위 화면 기준 작성 (ex. 로그인 화면, 상품 목록 화면 등)

#### 🧩 스토리보드 구성 항목 예시

- 화면 ID 및 이름
- 화면 개요 및 기능 설명
- UI 구성 요소 (버튼, 입력창 등)
- 데이터 흐름 및 전환 조건
- 비고 / 추가 설명

> 💡 "기획 흐름을 문서로 표현"하는 도구라는 점이 가장 자주 출제됨  
> 💡 기능 설명 + 화면 간 연계 흐름까지 포함하는 점에서 목업/와이어프레임과 구분됨

#### 📝 기출 포맷 예시

- 스토리보드의 목적은 무엇인가?
- 다음 중 스토리보드에서 표현되지 않는 정보는?
- 화면 설계 명세서에 포함될 항목으로 옳은 것은?

#### 🧠 용어 설명

- **스토리보드**: 화면 흐름과 구성 요소를 문서로 정리한 설계서
- **화면 설계 명세서**: 각 화면의 UI, 기능, 전환 조건 등을 포함하는 문서
- **시나리오 흐름**: 사용자가 기능을 수행할 때의 예상 경로 및 순서

## 🔹 195. 프로토타입 (Prototype)

#### 📘 정의
**프로토타입(Prototype)**은 실제 동작하는 수준으로 구현된 **인터랙티브한 화면 모형**으로,  
사용자의 피드백을 통해 **요구사항을 검증하고 UI/UX를 개선**하는 데 활용된다.  
정적 목업과 달리 **화면 간 전환, 이벤트 반응 등 실제처럼 체험 가능**하다.
페이퍼 - 아날로그식
디지털 - 프로그램 사용

#### 🧩 주요 특징

- **기능 구현 포함**: 클릭, 입력 등 사용자 상호작용 반영
- 사용자 피드백을 통해 **요구사항 미비점 조기 발견**
- 반복적인 설계 개선 가능 (인터랙션 중심)
- 개발 전 **UI 흐름과 UX 문제점 검증** 목적

#### 🧩 다른 도구와의 비교

| 구분 | 와이어프레임 | 목업(Mockup) | 프로토타입 |
|------|---------------|----------------|-------------|
| 표현 방식 | 구조 위주 | 시각 디자인 포함 | 실제 동작 구현 |
| 상호작용 | 없음 | 없음 | 있음 (클릭, 전환 등) |
| 활용 목적 | 기능 배치 구상 | 시각적 디자인 확인 | 사용성 테스트 및 요구 검증 |

> 💡 정처기 실기에서는 “프로토타입 = 동작하는 설계 모형”이란 키워드가 출제 핵심  
> 💡 목업과 차이를 구분하는 문제가 자주 나옴

#### 📝 기출 포맷 예시

- 프로토타입의 목적은 무엇인가?
- 다음 중 프로토타입의 특징으로 옳지 않은 것은?
- 목업과 프로토타입의 차이를 설명하시오.

#### 🧠 용어 설명

- **프로토타입**: 동작 가능한 인터페이스 모형
- **인터랙션**: 사용자의 입력에 따른 시스템 반응
- **사용성 테스트**: 실제 사용자가 시스템을 사용하면서 문제점을 찾아내는 과정

## 🔹 196. 유스케이스 (Use Case)

#### 📘 정의
**유스케이스(Use Case)**는 **시스템이 사용자와 상호작용하는 기능 단위 시나리오**를 의미하며,  
시스템의 **행위(기능)**를 사용자의 관점에서 **“무엇을 할 수 있는가”** 중심으로 표현한 것이다.  
요구사항 도출 및 기능 정의에 자주 활용된다.

#### 🧩 주요 특징

- 사용자(액터)와 시스템 간의 **상호작용을 시나리오 형태로 표현**
- 하나의 유스케이스 = 하나의 기능 수행 과정
- UML(Unified Modeling Language)의 대표 모델
- 다이어그램과 설명 문서 형태 모두 존재

#### 🧩 유스케이스 구성 요소

- **액터(Actor)**: 시스템과 상호작용하는 외부 주체 (사용자, 외부 시스템 등)
- **유스케이스(Use Case)**: 액터가 수행할 수 있는 기능 단위
- **시나리오(Scenario)**: 유스케이스 수행 절차 (기본 흐름, 예외 흐름 등)
- **시스템 경계(System Boundary)**: 시스템의 처리 범위를 명확히 구분

> 💡 유스케이스는 “사용자 관점의 기능 정의”라는 키워드로 자주 출제됨  
> 💡 유스케이스 = 요구사항 정리 및 설계 기반 자료

#### 📝 기출 포맷 예시

- 유스케이스 다이어그램의 구성 요소로 옳은 것은?
- 액터와 유스케이스의 관계를 설명하시오.
- 다음 중 유스케이스로 표현하기 적절한 것은?

#### 🧠 용어 설명

- **유스케이스**: 사용자와 시스템 간 상호작용 기반 기능 정의
- **액터(Actor)**: 시스템 외부에서 기능을 요청하는 주체
- **시스템 경계**: 시스템 내부 기능과 외부 인터페이스를 구분하는 선
- **시나리오**: 유스케이스 수행 절차의 흐름
